## 修复 Docker 部署报错 (minna-backend unhealthy)

### 根因分析
- 之前将 Dockerfile CMD 从 `prisma db push` 改为 `prisma migrate deploy`
- 但项目的迁移链**不完整**：init 迁移 (`20241212000000`) 只创建了 `Article` 表，其他表（`Craft`、`Idea`、`Goal` 等）没有在迁移文件中创建
- `merge_draft_into_article` 迁移引用了 `Draft` 表（`FROM "Draft"`），但全新数据库中该表不存在
- 导致 `prisma migrate deploy` 失败 → 后端无法启动 → healthcheck 失败 → frontend 依赖 backend healthy 也无法启动

### 修复方案（第三次迭代 — 最终版）
1. **新增 `docker-entrypoint.sh` 启动脚本**：取代 CMD 中的长命令链，更可靠且有详细日志
   - Step 1: `node migrate-data.js`（数据迁移，保护 Draft 数据）
   - Step 2: `npx prisma db push --skip-generate --accept-data-loss`（同步 schema，带重试）
   - Step 3: `exec node index.js`（启动服务）
2. **Dockerfile 优化**：
   - 添加 `apk add --no-cache openssl`（Prisma Alpine 依赖）
   - 改用 `npm install`（全量安装，不再 `--production`），避免 prisma CLI 安装问题
   - 移除冗余的 `npm install prisma --save-dev`
   - CMD 改为 `sh docker-entrypoint.sh`
3. **docker-compose healthcheck 调整**：
   - `start_period: 40s → 120s`（给 db push 更多时间）
   - `interval: 30s → 15s`（更频繁检查）
   - `retries: 3 → 5`（更多重试）

### 修复方案（第四次迭代 — Prisma Client 未初始化）
**问题**：容器启动报 `@prisma/client did not initialize yet. Please run "prisma generate"`
**根因**：
- Dockerfile 中 `prisma generate` 在 `COPY . .` **之前**执行
- `COPY . .` 把本地 backend 目录下的文件（可能包含 node_modules）覆盖了容器中已生成的 Prisma Client
- backend 目录下**没有 `.dockerignore`**（根目录的对 `build.context: ./backend` 不生效）

**修复**：
1. 创建 `backend/.dockerignore`，排除 `node_modules`
2. 把 `prisma generate` 移到 `COPY . .` 之后

### 变更文件
- `backend/Dockerfile`：重写（prisma generate 移到 COPY 之后）
- `backend/.dockerignore`：**新增**
- `backend/docker-entrypoint.sh`：**新增**启动脚本
- `backend/migrate-data.js`：数据迁移脚本（保留）
- `docker-compose.yml`：healthcheck 参数调整
